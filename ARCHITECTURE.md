# arstdhneio Architecture

## Overview
arstdhneio is a macOS utility that replaces mouse navigation with a keyboard-driven overlay. By default it activates through a registered `Cmd+;` hotkey, where the `;` binding follows the active layout's command-layer mapping rather than the fixed US semicolon key position. Keys refine a 4×10 grid that maps to the screen while warping the cursor to the refined centre, and pressing Space clicks without needing to move again. The package is split into a reusable core library and a macOS app target that hosts SwiftUI/AppKit windows for the overlay and zoom preview. The repo now also includes scripts that wrap the built executable into `arstdhneio.app`, which is the supported form for Launch at Login.

## Package layout
The Swift package exposes a platform-neutral `arstdhneioCore` library alongside the `arstdhneio` executable target. Core contains the state machine, grid math, input handling, and mouse action abstractions, while the executable hosts the SwiftUI/AppKit presentation that subscribes to core state and renders overlay windows on every screen. The test suite now covers both layers: most behavior tests target the core library directly, and a small app-target test bundle checks bundle metadata and launch-at-login state transitions.

## Core layer
- **Grid navigation**: `GridLayout` maps keyboard characters onto a grid and returns the subdivided rectangle for each key, enabling iterative refinement of the target region. `GridRect` provides geometric utilities including `center` for the midpoint, while `GridPartitioner` either splits columns across multiple displays for wider layouts or keeps narrow 5-column layouts intact on the screen under the cursor. The app can construct `GridLayout` from built-in presets such as `qwerty`, `colemak`, and the 4×5 `colemak5`, or from four custom row strings supplied at launch.
- **Overlay state and control**: `OverlayController` owns the overlay lifecycle. It starts the overlay with the active screen bounds, refines the current rectangle on key presses, and deactivates after a click or cancel. The controller maintains a history stack of previous states, allowing backspace to restore the previous level. The controller extracts clear helper methods: `selectScreenIfNeeded`, `refineGrid`, `applyRefinement`, and `zoomOut` make the key handling flow explicit. `OverlayState` holds the active flag, root bounds, refined rectangle, and derived target point so downstream consumers can react without duplicating business logic.
- **Input processing**: `InputManager` still owns the keyboard-to-overlay mapping, printable command-layer translation, and click/cancel commands. In the default `Cmd+;` activation mode, the app uses a Carbon global hotkey plus a focused overlay window for local key capture, which avoids the global input-monitoring event tap. The hotkey registration resolves the current layout's command-layer binding for `;` and re-registers itself when the selected keyboard input source changes. The older Double-Command mode is still available and routes modifier changes through `CommandTapRecognizer`, installing a CGEvent tap only for that higher-permission activation path. Printable bindings are derived through `CommandKeyMap`, which uses macOS keyboard-layout translation with the Command modifier applied so the overlay tracks the layout's shortcut-oriented Command layer rather than the base typing layer.
- **Pointer actions**: `MouseActionPerforming` abstracts cursor motion and clicking. The default `SystemMouseActionPerformer` can warp the cursor independently of clicks and post click events (left, middle, or right) at the current target, while tests can inject stubs to assert coordinates without real side effects.

## macOS app layer
- **App bootstrap**: `AppDelegate` wires the pieces together: it loads `AppConfiguration` from command-line arguments, environment variables, and persisted `UserDefaults` settings, instantiates the configured grid layout, overlay controller, and visual model, builds overlay windows per `NSScreen` with per-display grid slices, and starts either the global hotkey manager or the legacy input-monitoring event tap depending on the selected activation mode. It rebuilds windows when screens change so overlays and partitions stay in sync. Window visibility is managed through `updateWindowVisibility`, which shows/hides overlay windows based on state. The app creates a menu bar status item with About, Configuration, Launch at Login, and Quit options for easy access.
- **Overlay windows**: `OverlayWindowController` hosts an `OverlayGridView` inside a borderless `NSWindow` at the `.screenSaver` level. The SwiftUI view draws the translucent grid and highlights the current target rectangle for each display while ignoring mouse events. Each window receives its own `GridSlice` so labels and hit testing only cover the columns assigned to that screen. In `Cmd+;` mode, one overlay window temporarily becomes key so the app can capture refinement keys locally and then hand focus back to the previously active application.
- **State bridging for SwiftUI**: `OverlayVisualModel` mirrors core overlay state as an `ObservableObject`, letting SwiftUI views animate visibility and highlights as the controller emits updates. The app listens to `OverlayController.stateDidChange` and hops to the main actor with Swift concurrency tasks so UI updates stay in sync with the event tap callbacks while satisfying AppKit's main-thread isolation.
- **Bundle/runtime support**: `AppBundleSupport` centralizes the bundle metadata needed for `arstdhneio.app`, and `LaunchAtLoginManager` wraps `SMAppService.mainApp` so the menu bar can enable or disable login startup when the app is running from the actual `.app` bundle. `StoredAppSettingsStore` persists the activation mode, grid preset, and custom rows in `UserDefaults`, while the Configuration window lets the menu bar app edit those values and hot-reload the runtime when no launch-time override is active. Raw `swift run` executions intentionally keep Launch at Login unavailable because the login item belongs to the bundle, not the loose executable.

## Event flow
1. App launch initializes the controllers, windows, and screen observer, then starts the input manager.
2. Pressing `Cmd+;` toggles the overlay and shows the overlay windows. If the user switches to the legacy activation mode, Double-Command uses the CGEvent tap instead.
3. Each key press refines the grid rectangle via the core layout; the first key maps to a display when multiple screens are present, after which refinements stay within that slice, move the cursor to the refined centre, and update the highlight. The current state is pushed to history before each refinement.
4. Backspace pops the previous state from history, restoring the rectangle and cursor position to the previous level.
5. Arrow keys move the selected tile by half its width (left/right) or height (up/down), keeping its size unchanged while updating cursor position. Movement is blocked before a selection is made and constrained to stay within screen bounds.
6. Space posts a left-click at the refined target without warping again, after which the overlay hides; apostrophe (') posts a middle-click and backslash (\\) posts a right-click, both deactivating the overlay afterward; Escape cancels instead.

## Packaging and startup
- `scripts/build-app.sh` builds the release executable, creates `dist/arstdhneio.app`, writes `Contents/Info.plist` from the checked-in template, and applies ad-hoc signing when `codesign` is available. The generated bundle is a menu bar app (`LSUIElement=true`) with the stable bundle identifier `com.levyatsishin.arstdhneio`.
- `scripts/install-app.sh` copies the built bundle into `~/Applications/arstdhneio.app`, which is the simplest install location for local development without requiring admin privileges.
- CI uses the same script so manual artifacts and tagged releases publish a zipped `.app` bundle instead of a bare executable.
